## Overview

A modern C++23 project boilerplate using:

- [CMake](https://cmake.org) with Conan-generated presets
- [Conan 2](https://conan.io) for dependency management
- [Boost](https://www.boost.org) via Conan for portable utility libraries
- [GoogleTest](https://github.com/google/googletest) via Conan

## Prerequisites

- CMake 3.28+
- Conan 2
- A supported single-config CMake generator
  - Ninja
  - Unix Makefiles
- A compiler and standard library with working C++23 support
  - GCC 13+
  - LLVM Clang 17+
  - Apple Clang 17+ recommended

> [!IMPORTANT]
> If you are on a very new Apple Clang release, make sure your Conan installation and settings are
> up to date before running `conan profile detect`.

> [!NOTE]
> CMake presets are generated by Conan as part of the normal `make debug`, `make release`, and
> `make coverage` flows.

## Configure, build, and test

### Quick start

```console
git clone https://github.com/megabyde/cpp-boilerplate.git
cd cpp-boilerplate
make debug
```

To force a specific generator, set `CMAKE_GENERATOR` when configuring or building. For example:

```console
CMAKE_GENERATOR="Unix Makefiles" make debug
```

The generated CMake preset names are `conan-debug` and `conan-release`.

### Release build

```console
make release
```

`make debug`, `make release`, and `make coverage` are the normal entry points. The lower-level
`conan-%` and `configure-%` targets are only useful when you explicitly want to stop after
dependency resolution or CMake configure.

### Helper targets

To see all helper targets:

```console
make help
```

> [!NOTE]
> Configuring with `cmake --preset ...` or the Makefile configure/build targets refreshes the root
> `compile_commands.json` symlink to point at the active build directory.

The sample app also demonstrates a small Boost-powered helper that trims and joins
record fields before printing a summary line.

## Formatting and linting

```console
make format
make format-check
make lint
```

## Coverage

Generate an LCOV tracefile and HTML report with:

```console
make coverage
```

This writes:

- `build/debug/coverage.info`
- `build/debug/coverage-report/index.html`

> [!NOTE]
> Coverage reuses the `debug` build directory. The `debug` preset enables coverage
> instrumentation, while `release` remains a normal optimized build. `make coverage`
> reruns the debug test suite and then captures coverage from that build.

> [!NOTE]
> `make coverage` requires `lcov` and `genhtml` to be installed. On Ubuntu, the package is
> `lcov`. On macOS with Homebrew, use `brew install lcov`.

## Editor setup

### VS Code

The checked-in [.vscode/settings.json](.vscode/settings.json) is configured for the
preset-driven workflow:

- use CMake presets
- default to the `conan-debug` preset for configure/build/test
- use the root `compile_commands.json` symlink for C++ tooling

If you want to use Conan-generated presets in VS Code before running `make debug`, generate them
once with:

```console
make configure-debug
```

Then open the folder in VS Code and use the `conan-debug` or `conan-release` CMake preset.

### CLion

CLion should use the Conan-generated presets from `CMakeUserPresets.json` rather than a separate ad
hoc CMake profile.

Before the first configure in CLion, generate the Conan presets and toolchain files once with:

```console
make configure-debug
```

Then in CLion:

1. Open the project root
2. Select the `conan-debug` or `conan-release` preset as the active CMake profile
3. Reload CMake

> [!NOTE]
> This repo does not rely on the Conan CLion plugin. The existing Conan + preset flow is already
> the source of truth.

## Layout

- `include/`: public headers
- `src/`: application sources
- `tests/`: unit tests
- `conanfile.py`: Conan dependency definition
- `Makefile`: helper targets for local development and CI
